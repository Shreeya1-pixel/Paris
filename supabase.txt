# Openworld Paris — Supabase setup (beginner-friendly)

Follow these steps once. The app reads **only** from your Supabase project (no mock data). An empty database is valid: feeds and search simply return no rows.

---

## Step 1 — Create project

1. Open [https://supabase.com/dashboard](https://supabase.com/dashboard) and sign in.
2. **New project** → choose org, name, database password, region (EU recommended for Paris).
3. When the project is ready, open **Project Settings → API** and copy:
   - **Project URL** → `NEXT_PUBLIC_SUPABASE_URL`
   - **anon public** key → `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - **service_role** key → `SUPABASE_SERVICE_ROLE_KEY` (only for server scripts like `seed:paris`, never expose to the browser)

4. Create `.env.local` in the repo root:

```bash
NEXT_PUBLIC_SUPABASE_URL=https://YOUR_PROJECT.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_anon_key
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key
NEXT_PUBLIC_MAPBOX_ACCESS_TOKEN=your_mapbox_token
GEMINI_API_KEY=your_gemini_key
```

Restart `npm run dev` after changing env vars.

---

## Step 2 — Enable Auth

1. **Authentication → Providers**
   - Enable **Email**.
   - Enable **Google** (add OAuth client ID/secret from Google Cloud Console).
2. **Authentication → URL configuration**
   - **Site URL**: `http://localhost:3000`
   - **Redirect URLs**: add `http://localhost:3000/**` and your production URL when you deploy.

The app uses `@supabase/ssr` helpers. New users get a row in `public.users` via the `handle_new_user` trigger (see schema).

---

## Step 3 — Run SQL (tables, indexes, RLS)

In **SQL Editor**, run the full schema from `supabase/schema.sql` in this repository. It includes:

- `users` (profile: username, full_name, avatar_url, vibes, interests, arrondissement)
- `events` (title, i18n JSON, category, geo `lat`/`lng`, `start_time`, RLS: anyone can read active events, authenticated users insert their own)
- `paris_places` (curated places — seed with `npm run seed:paris` after setting service role)
- `saved_events`, `saved_places`, `event_attendees`
- Indexes on `events(lat,lng)`, `events(start_time)`, `events(category)`, etc.
- RLS policies aligned with the app (public read for active events and places)

**Minimal subset** (if you only want the tables from the spec — the repo’s `schema.sql` is richer and is the source of truth):

```sql
-- USERS
create table users (
  id uuid primary key references auth.users(id),
  username text unique not null,
  full_name text,
  avatar_url text,
  created_at timestamptz default now()
);

-- EVENTS
create table events (
  id uuid primary key default gen_random_uuid(),
  created_by uuid not null references users(id),
  title text not null,
  category text not null,
  start_time timestamptz not null,
  location_name text,
  arrondissement text,
  lat float8 not null,
  lng float8 not null,
  created_at timestamptz default now()
);

-- SAVED EVENTS
create table saved_events (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references users(id),
  event_id uuid not null references events(id),
  saved_at timestamptz default now(),
  unique(user_id, event_id)
);

create index on events(lat, lng);
create index on events(start_time);
create index on events(category);
```

Prefer running **`supabase/schema.sql`** so the app matches migrations and TypeScript types.

## Step 4 — RLS (important)

Example policies (adjust if you use the full `schema.sql`, which already defines policies):

```sql
alter table users enable row level security;
alter table events enable row level security;
alter table saved_events enable row level security;

-- users: public read usernames/profiles for social features (repo uses "viewable by everyone")
create policy "Users are viewable by everyone"
on users for select using (true);

create policy "Users can update own profile"
on users for update using (auth.uid() = id);

create policy "Users can insert own profile"
on users for insert with check (auth.uid() = id);

-- events
create policy "Active events are viewable by everyone"
on events for select using (status = 'active');

create policy "Auth users can create events"
on events for insert with check (auth.uid() = created_by);

-- saved_events
create policy "Users manage own saved events"
on saved_events for all
using (auth.uid() = user_id);
```

## Step 5 — Seed curated places (optional)

The **~50 Paris places** are loaded into `paris_places` via the seed script (not used as fake UI data — they are real DB rows). From the project root:

```bash
npm run seed:paris
```

Requires `SUPABASE_SERVICE_ROLE_KEY` and `NEXT_PUBLIC_SUPABASE_URL` in `.env.local`.

## API routes (this app)

| Route | Purpose |
|--------|---------|
| `GET /api/events/nearby` | `lat`, `lng`, `radius` (default 5 km, max 30 km), `limit`, `offset` |
| `GET /api/places/nearby` | Same idea for `paris_places` |
| `GET /api/discover` | Feed sections for Discover tab |
| `GET /api/search` | `q`, `category` / `categories`, `arrondissement`, `date_from`, `date_to`, `free_only`, `lat`, `lng` |
| `POST /api/places/insight` | Gemini blurbs for place detail (needs `GEMINI_API_KEY`) |
| `POST /api/chat` | Map chat bar (tier-1 local KB + Gemini) |

## Troubleshooting

- **503 “Supabase not configured”**: missing `NEXT_PUBLIC_SUPABASE_URL` or `NEXT_PUBLIC_SUPABASE_ANON_KEY`.
- **Empty map/discover**: no rows in `events` / `paris_places` — create an event (logged in) or run the seed script for places.
- **RLS errors on insert**: ensure you are logged in and policies match `created_by = auth.uid()` for events.
